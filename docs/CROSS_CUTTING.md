# 横断的な設計判断・未決事項・リスク（点検）

**ステータス:** 計画（実装未着手）
**役割:** 個別機能ドキュメントをまたぐ「**先に決めるべき／設計時に考慮／問題として把握**」を一覧化。機能をスムーズに連結するための土台。

凡例: 🔴 着手前に決める / 🟡 設計時に考慮 / ⚪ 把握しておくリスク

---

## A. 先に決めるべき（🔴 基盤になる判断）

1. 🔴 **メッセージの正準ID・重複排除**
   - 同一メールが複数フォルダ/アカウント/再同期で重複、`Message-ID` 欠落もある。
   - **正準キー**（`Message-ID` ＋ account、無い場合は from+date+本文 fingerprint）を**最初に決める**。flags・tags・`ai_annotations`・引用突合・スレッド所属が全てこれに紐づく。ここがブレると全機能が崩れる。

2. 🔴 **スレッド再構築アルゴリズムの“正”はどこに置くか（Rust か 共有TS mail-core か）**
   - デスクトップ Rust と モバイル(Expo) で**二重実装すると挙動が割れる**。
   - 推奨: **アルゴリズムは `packages/mail-core`(TS) に一本化**し、Rust は I/O・暗号・索引に専念（[CROSS_PLATFORM.md](CROSS_PLATFORM.md)）。Phase 4 前に確定。

3. 🔴 **論理スレッドIDの決定的生成＋再解析のべき等性＋手動上書きの保持**
   - 端末間は独立同期 → 同じ入力から**決定的に同じ論理スレッドID**が出る方式（ハッシュ等）にして自然収束させる。
   - 再解析しても**手動の分割/結合/再件名（`thread_assignment='manual'`）を壊さない**こと。

4. 🔴 **スキーマ・マイグレーション基盤を Phase 2 から堅牢に**
   - threading/filtering/AI/SNS でスキーマは大きく育つ。`user_version` ＋順序付きマイグレーションを最初から。後戻り不可の規律。

5. 🔴 **AI へ渡す経路を“単一の絞り”に集約（プライバシー不変条件）**
   - 「AIには伏字」を保証するには、**全AI呼び出しが `ai_redact_protected` を必ず通る**設計に。要約/分類/下書き/返信のどれか1経路でも素通しすると漏れる。最初から choke point を1つに。

6. 🔴 **鍵管理の全体設計**
   - SQLCipher DBキー / 添付暗号(AES-GCM) / keyring 資格情報・TSG Oneトークン / 保護領域(Lv1共有鍵→Lv2本人鍵)。
   - **DBキーの出所**（keyring 由来か・端末バインドか・パスフレーズか）を決める。**データルートを別ドライブへ移動**した時の復号可否（鍵は keyring＝端末側）に直結。保護領域ヘッダ `key=<鍵ID>` は Lv2 を見越して**前方互換な形**で v1 から確保。

## B. 設計時に考慮（🟡 後の手戻りを防ぐ）

7. 🟡 **「会話（conversation）」抽象を最初から汎用に**
   - いまメール論理スレッドと `sns_conversations` が別モデル。統合インボックスで合流させる予定。
   - SNS は後段でも、**email/SNS が共有する会話・インボックス抽象**を先に設計しておくと、SNS 着手時の大改修を回避できる。

8. 🟡 **連絡先＝結合のハブ（identity resolution）**
   - 「知り合い/取引実績」フィルタ、カレンダー参加者、SNS 差出人照合が全て `contacts` に join。
   - **アドレス/ハンドル/電話 → 人物**の名寄せ・マージ方針を早めに固める（filtering・calendar・SNS が依存）。

9. 🟡 **同期ウィンドウ境界の挙動**
   - ウィンドウ外の親メールが**ローカルに無い**＝引用突合・スレッドが部分的に。取引実績(双方向履歴)も同期分に限定。
   - 「親が無い時は部分スレッド表示＋必要時サーバー取得」を各機能で統一的に扱う（[SYNC.md](SYNC.md)）。

10. 🟡 **FTS5 の対象と再索引コスト**
    - 索引対象は `clean_body`（引用除去後）。**パーサ改善で clean_body が変わる→再解析・再索引**が必要。移行手順を想定。
    - 保護領域は**ローカルでは実値**を索引（自分のDB＝SQLCipher）。データルートが外部/可搬ドライブの場合の機密露出に注意（§13）。

11. 🟡 **型の共有戦略（ts-rs と Expo）**
    - 境界型は Rust の ts-rs 生成。モバイル(Expo)は Rust を持たない。
    - **生成した型を `packages/types` に publish し、デスクトップ/モバイル双方が consume**する一方向フローを決める。

12. 🟡 **オフライン/アウトボックス設計**
    - ローカルファースト。**送信キュー（オフライン時）**、フラグ変更の遅延反映、AI のオフライン挙動。送信・フラグ・AI に横断するので早めに。

13. 🟡 **派生データの暗号化範囲**
    - `ai_annotations`・FTS索引・**添付サムネ/cache** も機密を含む。DB は SQLCipher で守られるが、**`media`/`cache` が可搬ドライブ**にある場合の暗号化方針を決める。

## C. 把握しておくリスク（⚪ 落とし穴）

14. ⚪ **データルート可搬の失敗モード**: 外部ドライブ未接続で起動 → 読み取り専用/オフラインで穏当に劣化。別マシンへ移すと keyring 鍵が無く復号不可（端末バインド）。「可搬にするならパスフレーズ鍵」という選択をユーザーに提示するか要検討。

15. ⚪ **スレッド/引用パーサの実地多様性**: 日英の属性行・HTML/plain・各クライアント差。**実メールのテストコーパス**を早期に収集しないと精度が出ない。最大の技術的山場（[THREADING.md](THREADING.md)）。

16. ⚪ **保護領域は“提案”なので仕様の安定が重要**: `X-Protected-Regions` とマーカー記法は **v=1 でバージョン固定**し、後方互換を保つ（送信済みメールが将来壊れない）。

17. ⚪ **TSG One アカウント/トークン契約**: 取得・更新（refresh）・残量照会・使用量レポートの**API契約**を TSG One backend と先に握る。AI 機能と「共有保存」が依存。keyring 共通サービス名も確定。

18. ⚪ **プロバイダ事情（IMAP/SMTP）**: Gmail/Outlook はアプリパスワード(2FA)/OAuth 要求あり。基本方針（手動設定）は維持しつつ、**接続失敗時の分かりやすい誘導**（アプリパスワード手順）を用意。

19. ⚪ **性能予算**: 数万通での FTS・スレッド再計算・仮想スクロール・バックグラウンド同期が UI を止めない非同期設計（tokio）。大規模メールボックスでの初回同期/再解析の重さを見積もる。

20. ⚪ **送信パイプラインの合流点**: 「返信/新規」モード × 保護領域(伏字＋暗号) × AI下書き × スレッド整合 が**作成→送信の1パイプライン**できれいに合成されること。ヘッダ(Message-ID/References)生成の正しさ。

## 推奨アクション（順序）

1. Phase 2 で **正準ID＋マイグレーション基盤**（#1, #4）を固める。
2. Phase 4 前に **アルゴリズムの置き場所（mail-core）＋論理スレッドID方式**（#2, #3）を確定。
3. AI 着手前に **redaction choke point＋鍵管理＋TSG One契約**（#5, #6, #17）を設計。
4. SNS 着手前に **会話抽象＋連絡先名寄せ**（#7, #8）を用意。
5. 並行して **テストコーパス収集**（#15）を始める。
