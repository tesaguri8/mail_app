# 保護領域（プライバシー伏字）

**ステータス:** 計画（実装未着手）／差別化機能・**オープンな提案型の取り決め**
**目的:** メール本文中の機密部分を**伏字**で送り、実値は**どのメールクライアントでも開ける暗号コンテナ（暗証PDF）**で届ける。相手側の**不用意なクラウドAI処理に、プライバシーデータを自動で渡さない**ための仕組み。**将来の標準化のきっかけ**を狙う提案型（§10）。

関連: [AI_FEATURES.md](AI_FEATURES.md)（AIプライバシー方針）/ [FEATURE_SPEC.md](FEATURE_SPEC.md) / [THREADING.md](THREADING.md)

---

## 1. 目的と脅威モデル（割り切り）

- **守りたいもの**: 受信側のメールソフト/AIが本文・添付を**自動的に**クラウドAIへ送って処理する際の、機密の漏えい。
- **狙うのは「事故防止」**: 完璧な防御ではなく、**うっかり自動処理でAIに渡る**のを防ぐ。
- 本文に書いたパスワードを手動でAIに食わせ、添付まで開かせる相手は**対象外**（そこまでする相手に防御コストを払わない）。
- **確実な防御は暗号化**。伏字・記法はUXと構造であって防御ではない（暗号コンテナが防御）。

---

## 2. 記法（本文中の伏字マーカー）

```
/////privacy_address/////
```

- 区切り `/////` ＋ **任意のフィールド名** ＋ `/////`。
- プレーン/HTML どちらでも壊れにくい中立記法（タグ風 `<...>` は HTML で問題になり得るため不採用）。
- どのクライアントでも**ただの伏字テキスト**として表示される。

---

## 3. 二段構え（提案の本体＝ヘッダ方式 ＋ フォールバック＝PDF）

| 経路 | 位置づけ | 中身 |
|---|---|---|
| **ヘッダ方式の暗号化保護領域**（AES-256-GCM） | **提案の本体（相互運用）。** Rondine 同士、および本提案を実装した任意クライアントで成立 | **独自ヘッダ**にメタ情報＋暗号データ（小）／大きい場合は専用 MIME パートを参照。受信者の**公開鍵で暗号化**（自動鍵交換） |
| **暗証番号付き PDF**（AES-256） | **フォールバック（非対応クライアント向け）。** 全OS/全ビューアがパスワード対応＝誰でも開ける | 実値を暗証PDFに格納、パスワードは本文表示 |
| 暗号ZIP | 不採用（OS標準がAES非対応/弱いZipCrypto） | — |

- **提案として広めたいのはヘッダ方式**（軽量・相互運用・参照実装で動かせる）。これが標準化の核。
- **非対応クライアント相手でも破綻しない**よう、PDF フォールバックを併用（「どのクライアントでも内容にアクセスできる」を担保）。
- どちらを使うかは「相手が本提案に対応しているか」で**自動選択**（過去のやり取り/鍵の有無で判定）。

### ヘッダ方式の形（提案・ベンダー中立）

```
# 本文（どのクライアントでも伏字テキストとして見える）
ここは /////privacy_address/////  です。

# ヘッダ（メタ情報＋暗号データ。実値の平文は載せない）
X-Protected-Regions: v=1; regions=privacy_address;
    alg=AES-256-GCM; enc=base64; key=<受信者鍵ID>; data=<base64暗号文>
#   ※ 暗号文が大きい場合は data の代わりに part=<Content-ID> で MIME パートを参照
```

> ヘッダ名はベンダー中立（例 `X-Protected-Regions`）。Rondine はその参照実装として動く。

## 3.5 段階的セキュリティレベル（MVP から上げていく）

「動く参照実装を最速で」出すため、**まず最小から始めて段階的に強化**する。

| レベル | 鍵の持ち方 | 解読できる人 | 位置づけ |
|---|---|---|---|
| **Lv1（MVP・最初の一歩）** | **Rondine がアプリ共有鍵を保持** | **Rondine を持つ人**（鍵が無ければ解読不可） | 最小実装・**鍵交換不要**。AI と非対応クライアントから隠せる（事故防止レベル）。参照実装を即出せる |
| **Lv2** | **ユーザー個別の鍵ペア＋自動鍵交換**（公開鍵暗号） | **宛先本人のみ** | アプリ鍵流出に強い。本人だけが復号。パスワードレス |
| **Lv3** | メッセージ毎パスワード／別経路、PDFフォールバック | パスワードを知る人 | 非対応クライアント対応・任意の追加強度 |
| **Lv4（任意）** | S/MIME・PGP 連携 | 標準E2E | 既存標準と相互運用したい上級者向け |

> **正直な注意（Lv1）**: アプリ共有鍵は、アプリからの鍵抽出には弱い（＝強固なE2Eではない）。しかし目的が「**不用意なAI露出の防止**」である限り十分機能し、**参照実装を素早く成立**させられる。安全性は Lv2 以降で本人鍵へ引き上げる。

**着手順**: Lv1（共有鍵）で仕組みを通す → Lv2（本人鍵・自動鍵交換）で安全性を上げる → Lv3/Lv4 を選択肢として追加。

## 4. 送信フロー

0. **保護領域が1つも無ければ、暗号化も PDF も付けず“普通のメール”として送信**（`X-Protected-Regions` ヘッダも付与しない）。以降は**保護領域がある場合のみ**。
1. 作成画面で機密スパンを選択 → 各に**任意のフィールド名**（`privacy_address` 等）を付与。
2. 本文はその位置を **`/////privacy_address/////`** で伏字化して送信。
3. 実値（フィールド名→値）を暗号化:
   - **相手が本提案に対応**（公開鍵あり）→ **ヘッダ方式**（受信者の公開鍵で AES-256-GCM 暗号化し `X-Protected-Regions` ヘッダ／MIME パートに格納）。鍵方式なのでパスワード不要。
   - **非対応の可能性** → **暗証PDF** を添付。**ランダムなパスワードで暗号化（AES-256）し、その暗証番号を本文へ自動追記**する。
   - 両方併載も可（対応クライアントはヘッダ、非対応はPDF）。
4. **暗証番号の自動追記**（PDF を付ける場合）:
   - 送信時にメッセージ毎の**ランダムパスワードを生成 → PDF を暗号化 → 本文末尾等に定型文で自動挿入**（例: 「添付PDFのパスワード: ****」。i18n 対応）。
   - **送信前プレビュー（§5.5）で確認・編集可能**。固くしたい人は「本文に書かず別経路で渡す」をオプション選択。

---

## 5. 受信フロー

- **本提案に対応（Rondine 等）**: `X-Protected-Regions` を検出 → 鍵で復号 → **伏字位置に実値をインライン表示**（“随時表示”）。
- **非対応クライアント**: 伏字本文＋本文中のパスワードで**添付PDFを開く** → 実値を読める（破綻しない段階的劣化）。
- **相手のAIから見えるもの**: 伏字本文＋（開けない暗号データ/PDF）＝ **本物は渡らない** ✅

## 5.5 表示トグル（伏字⇄実値）と送信前確認

本文表示は **伏字／実値をワンタッチで切り替え**できる。安心して送れるための要。

- **送信前プレビュー（安心の核）**: 作成画面で「**相手・AIに渡る姿（伏字）**」と「**実値**」をトグルで確認。
  - 保護領域は色付きでハイライト。**何が隠れ、何が平文で出るか**を送信前に目視チェック。
  - 「保護し忘れ」が無いかも確認できる（伏字側に出ていない機密＝そのまま送られる）。
  - 送信ボタン付近に「送信プレビュー：伏字 ON」を常時表示し、**今どちらを見ているか**を明確に。
- **受信表示**: 「伏字」⇄「実値（復号）」をトグル。
  - 既定は設定可（人前で開く時は伏字のまま、など）。Rondine は信頼端末として実値表示も可。
- **AIビューの明示**: 「AIにはこう見える」を同じ伏字ビューで確認できる（[AI_FEATURES.md](AI_FEATURES.md) の伏字置換と一致）。

> ねらい: 「ちゃんと伏字になっている」を**送る前に自分の目で確認**できる安心感。

## 6. 核となる動作ルール（全機能で徹底）

> **保護領域は「人には実値・AIには伏字」。**

- Rondine 自身のAI機能（要約・分類・下書き・[AI_FEATURES.md](AI_FEATURES.md)）に渡す前に、**保護領域を伏字へ置換**してから送る。
- 受信表示は人には実値（復号後）、しかし**AI処理時は伏字**。
- 送信本文は伏字、実値は暗号コンテナ。
- これにより「うっかりクラウドAIに本物が流れる」事故を**自動で**防止。

## 7. 鍵・パスワード方針

- **PDF**: パスワードは本文表示（既定）。オプションで別経路（SMS等）。
- **軽量パート（Rondine 同士）**: 相手の公開鍵で暗号化＝**自動鍵交換でパスワードレス**（将来。最も滑らか）。

## 8. 守れること／守れないこと（正直に）

- ✅ 受信側AIの**自動処理**から機密を隠せる（暗号データ/暗号PDFは読めない）。
- ✅ 実値へのアクセス: 対応クライアントは復号表示／非対応クライアントは**PDFフォールバック併載時**に開ける（普遍性）。
- ⚠️ **Lv1（アプリ共有鍵）はアプリからの鍵抽出に弱い**（強固なE2Eではない）。事故防止には十分。安全性は Lv2（本人鍵）で引き上げる（§3.5）。
- ⚠️ PDFのパスワードを本文に書く場合、**狙って**復号する高度エージェントAIは理論上開ける（脅威モデル外）。
- ⚠️ 「機密が存在する事実」と前後文脈は残る（通常許容）。
- 🔒 より固くするなら：本人鍵（Lv2）／パスワード別経路／S/MIME・PGP（Lv4）。

## 9. 実装メモ

- **MVP（Lv1）が最小・最速**: ヘッダ方式＋**アプリ共有鍵**＋ **`aes-gcm`**（計画済みクレート）だけで成立。PDF も鍵交換も不要 → **参照実装を即出せる**。
- **Lv2 以降**: ユーザー鍵ペア＋自動鍵交換（本人のみ復号）。
- **暗号化PDF生成（フォールバック・後続）**: Rust の多くの PDF クレートは暗号化未対応 → **qpdf 等が必要**。AES-256 指定（旧 RC4 不可）。MVP では不要、非対応クライアント対応が要る段で追加。
- 受信時の検出: `X-Protected-Regions` ヘッダ＋本文マーカー（`/////name/////`）で判定。スキーマ変更は最小（必要なら `emails.has_protected_regions` フラグ程度）。
- アプリ共有鍵の保管: 平文同梱でなく、難読化／ビルド時注入など可能な範囲で保護（Lv1 は“事故防止”が目的のため完全秘匿は前提にしない）。
- **鍵の世代管理（重要・前方互換）**: 焼きこみ共有鍵に**世代ID**を付け、ヘッダの `key=<鍵ID>`（例 `appkey-v1`）に記録。後で鍵を差し替え／Lv2（本人鍵）へ進化しても、**過去に送ったメールを引き続き復号**できる。研究開発で“使いながら育てる”ための保険。

> 開発スタンス: Lv1（焼きこみ共有鍵）で素早く動かし、**使いながら反復（研究開発）**で Lv2 以降へ。鍵世代IDだけ最初から入れておけば、後戻りなく強化できる。

### Tauri コマンド（抜粋）
| コマンド | 用途 |
|---|---|
| `compose_add_protected_region` | 作成時に機密スパンを保護領域化（伏字＋実値登録） |
| `compose_build_protected` | **保護領域がある場合のみ**、伏字本文＋暗号(ヘッダ/パート)＋（必要時）暗証PDFを生成し、**PDFパスワードを本文へ自動追記**。保護領域ゼロなら通常メールをそのまま返す |
| `message_reveal_protected` | 受信メールの保護領域を復号してインライン表示 |
| `protected_preview_masked` | 送信前/受信時に「伏字で渡る姿」をレンダリング（表示トグル用） |
| `ai_redact_protected` | AI へ渡す前に保護領域を伏字へ置換（全AI機能が経由） |

## 10. オープンな提案として（標準化の種）

本機能は**特定アプリ専用の囲い込みではなく、提案型のオープンな取り決め**として設計する。普及するほど実効性が増す性質のため、誰でも実装できるよう仕様を公開する。

- **最小で誰でも実装できる**こと（記法 `/////name/////` ＋ `X-Protected-Regions` ＋ AES-256-GCM）を重視。複雑な合意なしに片側だけでも成立する（段階的劣化）。
- **MVP から始める**: まず Lv1（アプリ共有鍵）で**動く参照実装**を出し、Lv2（本人鍵）以降で安全性を上げる（§3.5）。「動く実装」があって初めて提案になる。
- **既存の近い仕組みに乗る**: 慣習ヘッダ `Sensitivity:`（Personal/Private/Confidential）や、Webで進む「AI利用の意思表示（AI preferences 等）」と思想を合わせ、孤立した独自仕様にしない。
- **コア原則をシンプルな一文に**:「**保護領域は人には実値・AIには伏字**」。この一行を他クライアントが尊重するだけで広がる。
- **普及の道筋**: ①Rondine がリファレンス実装として動く → ②記法・ヘッダ・コンテナ仕様を文書公開 → ③賛同クライアントが「AIには伏字」を実装 → ④デファクト化 → ⑤必要なら標準化提案（IETF 等）。
- 目的は「**不用意なAI露出を、業界全体で減らす**」こと。Rondine はその**きっかけ（提案者）**を担う。

> ねらい: 勝つための独自仕様ではなく、**広めるための提案**。シンプルさと段階的劣化が普及の鍵。

> **公開の切り分け**: 相互運用に必要な部分だけを**ベンダー中立の公開仕様** [`spec/protected-regions-v1.md`](../spec/protected-regions-v1.md) に切り出す。アプリ本体コードは非公開、この仕様だけ公開（[POSITIONING.md](POSITIONING.md) ライセンス・公開方針）。本書（`docs/`）はアプリ内部設計、`spec/` は公開面。

## 10.5 配送性・互換性・セキュリティ装置との関係（重要）

独自ヘッダや暗号添付を載せても**メールが壊れない／消えない**ことを設計で担保する。

### 他クライアント・サーバーに削除されるか
- **メールクライアントは削除しない**（未知ヘッダは無視するだけ。メールは丸ごと保持）。
- **大半の MTA は素通し**（規格上、中継は本文・ヘッダを改変しない建前）。
- **例外（100%保証ではない）**: メーリングリスト（ヘッダ書換/フッター付与）、企業 Exchange/ゲートウェイの transport rule による独自ヘッダ除去。
- **対策**: 正しさを**ヘッダ単独に依存させない**。
  - 暗号データは**本文の MIME パート**に置く（本文側はヘッダより残りやすい）。ヘッダは**メタ＋鍵IDだけ**。
  - **PDF フォールバック**を併用。
  - **伏字本文（普通のテキスト）は必ず残る**＝最悪でもメールは壊れず伏字のまま届く（段階的劣化）。

### セキュリティ装置にひっかかるか
- **独自ヘッダ自体はほぼ無害**（正規メールも X- ヘッダ多数）。**SPF/DKIM/DMARC にも無関係**（署名を壊さない限り）。スパムスコアにも基本影響しない。
- **要注意は暗号化された添付（パスワードPDF／不透明 blob）**: 中身をスキャンできない暗号添付を**ブロック/隔離する企業ゲートウェイ**がある（マルウェア対策）。DLP がポリシーで止めることも。**ここが最もひっかかり得る**。
- **対策**: 暗号は MIME パート基本、PDF は「相手組織によっては暗号添付をブロックし得る」と UI で注意表示。DKIM は普通に署名し不自然な難読化を避ける。主要プロバイダ（Gmail/Outlook/Exchange）で**配送テスト**。

### 残存性とセキュリティ（まとめ）
| 要素 | 残存性 | セキュリティ |
|---|---|---|
| 伏字本文（普通テキスト） | ◎ 必ず残る | 無害（安全な土台） |
| ヘッダ（メタ＋鍵ID） | ○ ほぼ残る（稀に剥がれる） | ほぼ無害 |
| 暗号 MIME パート | ○ 残りやすい | 不透明 blob は稀に警戒 |
| パスワードPDF（フォールバック） | ○ 添付は残る | **暗号添付ブロックに注意** |

> 設計の効き目: **「伏字本文だけは絶対届く」を土台**に、暗号部分が一部経路で消えても**メール自体は破綻しない**（段階的劣化）。

## 11. 鍵のバージョン管理（メールとの関係・ローテーション）

**結論: 切替を「日付/期間」で分けない。各メッセージが“使った鍵ID”を名乗る（自己記述）。**

- **日付方式は危険**: クロックずれ・TZ・送信遅延・再送・下書きで境界が曖昧になり、どの鍵か特定できない事故が起きる。「いつ切り替えたか」の別管理表もずれる。
- **自己記述方式（採用）**:
  1. 各保護メッセージは**使った鍵IDをヘッダに記録**（`key=appkey-v1`）。復号は「メッセージが名乗る鍵ID → 対応する鍵」で**決定的**に行う（時刻判定なし）。
  2. アプリは**鍵レジストリ** `{ 鍵ID → 鍵素材 }` を保持。ローテーション＝**新しい鍵を足して「現在の暗号用」ポインタを切替えるだけ**。新規は現行鍵、過去は自分の鍵IDのまま復号可。**切替時期の曖昧さが原理的に発生しない**。
  3. **旧鍵は消さない**（消すと過去メールが読めない）＝唯一の鉄則。
- **メールデータとの関係**: 紐付けは**メッセージ単位**（ヘッダ `key=<鍵ID>` が真実の源）。他クライアント・他端末でも成立。ローカルDBにも各メールの鍵IDを控えると検索・一括再暗号化が速い（正はヘッダ）。
- **世代＋用途で Lv2 まで一貫**: `appkey-v1`（Lv1共有）→ `userkey-<id>-v1`（Lv2本人鍵）。同じ自己記述の仕組みで進化。
- 鍵漏えい時: 新鍵へローテート（新規は新鍵）、旧鍵は「復号専用・要注意」として保持。日付方式では表現しづらいが、鍵ID方式なら自然に扱える。
- **鍵にスタート日時等のメタを付けてよい（ただし“記録”であって“選択”ではない）**:
  - レジストリ例: `{ "appkey-v1": { material, created_at, status: active|retired|compromised, current? } }`。
  - 用途は**運用・監査・現行鍵の選択**（新規暗号化は `current` の鍵を使う。日時は補助）。
  - **復号の鍵選択には日時を使わない**（常にメッセージの鍵ID）。日時を選択ロジックに入れると“切替の曖昧さ”が再発する。
  - 任意の整合チェック: 「鍵IDの導入日時より送信が前」なら改ざん/時計ズレの警告に使える（検知用。復号は鍵IDのまま）。
