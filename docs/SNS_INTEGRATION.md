# SNS 統合（メッセージハブ）計画

**ステータス:** 計画（実装未着手）／**後続ステップ — コア機能の安定後に着手**
**位置づけ:** 本アプリを「メール / SNS DM / コメント」を 1 箇所に集約する **メッセージハブ** とするための設計。
**着手タイミング:** **まずコア機能（メール＋住所録＋カレンダー, Phase 1〜9）を安定させることを最優先**とし、SNS 統合はその次の段階として実装する。本書はそのための先行設計（可否・アーキテクチャの整理）。
**動機:** 複数 SNS のやり取りがバラバラだと見落とす。特にふくぎリビングのような宿泊施設では、問い合わせへの気づきの遅れが機会損失・信頼低下に直結する。すべてを 1 つのチャット形式インボックスに束ね、取りこぼしをなくす。

## 1. コンセプト：メッセージハブ

```
   ┌── メール（IMAP/SMTP）
   ├── LINE 公式アカウント
   ├── Instagram（DM + コメント）
   ├── Facebook Messenger（ページDM + コメント）
   └── WhatsApp Business
                   │  すべてを共通スキーマに正規化
                   ▼
        ┌──────────────────────────┐
        │  統合インボックス（ハブ）  │  ← チャット形式ビューで一元表示
        │  未読・担当・返信を一元管理 │
        └──────────────────────────┘
```

- チャネルが違っても、UI は同じ「会話（スレッド）×メッセージ」で扱う。既存のチャット形式ビュー（[UI_UX_DESIGN.md](UI_UX_DESIGN.md) §4）をそのまま再利用できる。
- ホーム/ウィジェットには **全チャネル合算の未読数**と最新の問い合わせを表示し、「気づきの遅れ」をなくす。

### メールが SNS に代替されない理由（本質的な違い）

SNS DM を束ねても、**メールは中核として残る**。SNS が構造的に代替できない、メール固有の 2 つの強みがあるため：

1. **記録性（永続性・可搬性）** — メールは自分の手元（ローカル DB）に**恒久的に残せる証跡**。過去のやり取りを検索・エクスポート・保管でき、契約・予約・請求など「後から証拠として遡れる」性質を持つ。SNS の履歴は**プラットフォーム側に預けたもの**で、アカウント凍結・仕様変更・サービス終了で失われうる（＝借り物）。
2. **アプリ/事業者に囲い込まれない（オープン標準）** — メールは IMAP/SMTP という**ベンダー中立の公開規格**。どのクライアント・どの事業者とも相互運用でき、相手が同じアプリを使っている必要がない。SNS は**各社の閉じた API・規約に縛られ**（24時間ウィンドウ・審査・API 廃止等）、相手も同じプラットフォームに居ることが前提になる。

> つまり本アプリの立ち位置は「SNS でメールを置き換える」ではなく、**記録性とオープン性を持つメールを土台に、閉じた SNS の入口だけを束ねる**こと。この非対称性が「メッセージハブ」の設計思想の核であり、逆にメール側の**ローカル完結・可搬性（[IMPORT_EXPORT.md](IMPORT_EXPORT.md)）**を守る理由でもある。

## 2. プラットフォーム別の実現性

| プラットフォーム | DM | コメント | 方式 | 必要条件・制約 |
|---|---|---|---|---|
| **LINE 公式アカウント** | ✅ | （公開コメントなし） | Messaging API（Webhook受信 / reply・push送信） | LINE公式アカウント + Messaging APIチャネル。比較的容易・低コスト |
| **Instagram** | ✅ | ✅（自投稿） | Meta Graph API（Webhook） | **プロアカウント + FBページ連携 + アプリ審査 + ビジネス認証**。権限 `instagram_manage_messages` / `instagram_manage_comments`。24時間返信ウィンドウ |
| **Facebook Messenger** | ✅ | ✅（自ページ投稿） | Messenger Platform（Webhook） | ページ + アプリ審査（`pages_messaging` / `pages_read_engagement`）。Instagram と Meta 基盤を共有 |
| **WhatsApp Business** | ✅ | — | Cloud API（Meta, Webhook） | WhatsApp Business Account + 電話番号。24時間ウィンドウ外はテンプレート審査が必要 |

### スコープ外・埋められない穴（正直な制約）
- **X (旧Twitter)**: DM は API v2 の有料プランが必要・レート制限厳。費用対効果を見て将来判断。
- **TikTok**: ユーザー DM を読む公開 API が存在しない → 対応不可。
- **Airbnb（個人ホスト）**: 公開メッセージング API なし。**Booking.com** はパートナー API で可能性あり（SNSではないが宿泊業の主要窓口として将来検討）。
- **規約遵守**: スクレイピングは BAN・法的リスクのため行わない。**公式 API のみ**を使用する。

## 3. アーキテクチャ（最重要）

### 3.1 なぜクラウド中継が必要か

これらの API は大半が **Webhook（プッシュ）型**で、新着を受け取るには **公開された固定 HTTPS エンドポイント**が要る。デスクトップアプリ（Tauri）単体では受信できない。加えて:

- 各プラットフォームの**シークレット/トークンを全端末に置きたくない**（漏洩リスク）
- **レート制限・24時間ウィンドウ**を一元管理したい
- 複数端末・将来のモバイルからも同じ受信箱を見たい

→ **小さなクラウド中継サービス**を設け、そこで受信・検証・正規化・保管し、アプリへ配信する。

> **設計上のトレードオフ**: メール部分は引き続きローカル完結（Tauri + Rust）だが、SNS 統合は**クラウド中継を前提とする**。これは「単一ローカルアプリ」方針からの意図的な例外。tesaguri-tech のバックエンド基盤（FastAPI + nginx プロキシ）に相乗りすることで新規インフラを最小化する。

### 3.2 構成図

```
各SNS ──Webhook──▶ ┌─────────────────────────────┐
（署名付き）        │  SNS 中継サービス（クラウド）  │
                   │  1. 署名検証                  │
                   │  2. 共通スキーマへ正規化       │
                   │  3. 保管（DB）               │
                   │  4. アプリへ配信（WebSocket）  │
                   └───────────┬─────────────────┘
                               │  認証付き WebSocket / ポーリング
                               ▼
                   ┌─────────────────────────────┐
                   │  Tauri アプリ（デスクトップ）  │
                   │  統合インボックス表示・返信操作 │
                   └─────────────────────────────┘
   送信: アプリ ──▶ 中継サービス ──▶ 各SNSの送信API（トークンはサーバー保管）
```

- **受信**: 各プラットフォーム → 中継（Webhook）→ 正規化 → アプリへ push。アプリがオフラインの間は中継が保管し、再接続時に差分配信。
- **送信**: アプリ → 中継 → 各 API。**トークン・シークレットはサーバー側にのみ保持**。
- **ローカルキャッシュ**: アプリは受信メッセージをローカル DB（SQLite）にもキャッシュし、オフライン閲覧・横断検索（FTS5）を可能にする。

### 3.3 既存基盤との関係
- 中継サービスは tesaguri-tech バックエンド（FastAPI）に新モジュールとして実装、`main-proxy`（nginx）配下に Webhook 受信パスを公開。
- 施設（例: ふくぎリビング）ごとに接続アカウント（チャネル）を登録・管理。

## 4. 共通メッセージスキーマ（正規化）

チャネル差を吸収する正規化モデル。詳細テーブルは [DATABASE_SCHEMA.md](DATABASE_SCHEMA.md) を参照。

```jsonc
{
  "channel": "line | instagram | messenger | whatsapp | email",
  "kind": "dm | comment",
  "direction": "inbound | outbound",
  "conversation_key": "チャネル内の会話/スレッド識別子（投稿IDやユーザーID）",
  "external_message_id": "プラットフォームのメッセージID",
  "sender": { "id": "...", "name": "...", "handle": "..." },
  "body_text": "本文（テキスト正規化）",
  "attachments": [{ "type": "image|file|...", "url_or_ref": "..." }],
  "timestamp": "ISO8601",
  "status": "unread | read | replied",
  "raw_ref": "元ペイロードへの参照（監査・再処理用）"
}
```

- 連絡先（住所録）との突き合わせ: 送信者のメール/電話/ハンドルで `contacts` にマッチし、人物単位で会話を束ねる将来拡張も可能。

## 5. 機能要件

- **統合インボックス**: 全チャネル横断の一覧。チャネルアイコンで出自を明示。チャット形式ビューで会話表示。
- **未読・担当・対応状態**: 未読バッジ（ホーム/ウィジェットに合算表示）、対応済みフラグ、担当者メモ（将来のチーム共有）。
- **返信**: 各チャネルの制約（24時間ウィンドウ等）を UI で明示し、可能な範囲で直接返信。
- **コメント対応**: Instagram/Facebook の自投稿コメントを取得・返信（チャネル種別 `comment`）。
- **通知**: 新着 SNS メッセージのデスクトップ通知。重要キーワード（「予約」「キャンセル」等）の強調。
- **横断検索**: ローカルキャッシュに対し FTS5 で全チャネル検索。

## 6. フェーズ計画（SNS 統合トラック）

**前提: コア機能（メール＋住所録＋カレンダー）が安定してから着手する。** 基本機能の安定を最優先とし、その後に下記を順に進める。

1. **S1 — 基盤**: 共通スキーマ確定、クラウド中継サービス雛形、アプリ⇄中継の認証付き WebSocket、ローカルキャッシュ DB。
2. **S2 — LINE**: 最も容易で日本の宿泊業に最重要。受信・返信・通知まで一気通貫。
3. **S3 — Meta（Instagram + Messenger）**: 共通の Meta 基盤で DM + コメント。アプリ審査・ビジネス認証を並行申請。
4. **S4 — WhatsApp Business**: 海外ゲスト向け。テンプレート審査対応。
5. **S5 — 運用強化**: 担当・対応状態、キーワード強調、複数施設管理、（将来）Booking.com・X の検討。

## 7. リスクと留意点

| 項目 | 内容 | 対応 |
|---|---|---|
| 審査・本人確認 | Meta/WhatsApp はアプリ審査・ビジネス認証に時間（数日〜数週間） | 早期に申請着手。LINE から先行 |
| 24時間ウィンドウ | Meta/WhatsApp は顧客最終発言から24時間外は自由返信不可（テンプレ必要） | UI で残り時間表示、テンプレ管理 |
| API 仕様・料金変更 | 各社ポリシー/価格は変動が速い | 実装時に最新の公式ドキュメントで再確認（本書は計画時点の整理） |
| トークン管理 | アクセストークン失効・更新 | サーバー側で集中管理・自動リフレッシュ |
| プライバシー/保管 | ゲストの個人情報を中継・保管 | 暗号化保管、最小権限、保持期間ポリシー、規約・法令順守 |
| 単一アプリ方針からの逸脱 | SNS はクラウド依存 | メールはローカル完結を維持し、SNS のみ中継前提と明確に分離 |

## 8. まとめ（技術的可否の結論）

- **LINE / Instagram(DM+コメント) / Messenger / WhatsApp は公式 API で実現可能。**
- 鍵は **クラウド中継サービス**（Webhook 受信・正規化・配信）を設けること。tesaguri 基盤に相乗りできる。
- **TikTok の DM・Airbnb 個人窓口は不可**など、埋められない穴は正直に残る。
- 正規化すれば既存のチャット形式 UI・FTS5 検索・ホーム/ウィジェットの未読表示にそのまま乗る。
