# 同期範囲・保持期間（取得する期間の選択）

**ステータス:** 計画（実装未着手）
**結論:** **すべてを無条件に取り込むのではなく、取得する期間をユーザーが選べる**（アカウント単位）。本文・添付は段階的に取得して軽快さと容量を両立する。

関連: [DEVELOPMENT_PLAN.md](DEVELOPMENT_PLAN.md)（Phase 4）/ [CROSS_PLATFORM.md](CROSS_PLATFORM.md)（独立 IMAP 同期）/ [DATABASE_SCHEMA.md](DATABASE_SCHEMA.md) / [DATA_STORAGE.md](DATA_STORAGE.md)

---

## 1. 同期ウィンドウ（ユーザー選択）

アカウントごとに「どこまで遡って取り込むか」を選べる。

- 選択肢（例）: **過去 1ヶ月 / 3ヶ月 / 6ヶ月 / 1年 / 2年 / 全期間**
- **既定: 6ヶ月**（軽快さと実用性のバランス。`全期間` はオプトイン）
- 端末ごとに異なる設定でよい（[CROSS_PLATFORM.md](CROSS_PLATFORM.md) の独立 IMAP 同期と整合。各端末が自分のウィンドウで同期）。
- 変更時: ウィンドウを広げたら不足分を追加取得、狭めたら範囲外をローカルから整理（下記 保持ポリシー）。

---

## 2. 段階的取得（メタデータ → 本文 → 添付）

全文を一度に落とさず、必要なものから取得して軽量・高速に保つ。

| レイヤ | 取得方針 |
|---|---|
| **ヘッダ/メタデータ** | 同期ウィンドウ内は取得。スレッド再構築・フィルタ・一覧の基盤（軽い） |
| **本文（clean_body）** | ウィンドウ内は取得（既定）。または開いた時にオンデマンド取得も選択可 |
| **添付ファイル** | **遅延取得**（開いた時／明示ダウンロード時）。サイズ上限・自動DL条件は設定可 |

> これにより「メタデータは広く・本文/添付は必要分だけ」を実現し、初回同期と容量を抑える。

---

## 3. 保持ポリシー（retention）

ローカルにどこまで残すかを選べる。

- **`window`（既定）**: 同期ウィンドウ内のみ保持。範囲外はローカルから整理。
- **`metadata_only`**: 範囲外も**メタデータ（ヘッダ・索引）は保持**し、本文/添付だけ削除して容量節約（検索ヒットは残る）。
- **`keep_all`**: 取得したものは消さない（容量優先でないユーザー向け）。
- 整理した本文/添付は**必要時に IMAP サーバーから再取得**（サーバーが源 → ローカルは可変キャッシュ）。

---

## 3.5 ローカル容量のグラデーション保存（3ティア）

**ステータス:** 日数ベースは実装済み（`feature/storage-tiers`）／%グラデーションは設計案として保持。

**基本の考え方＝ローカルはキャッシュ。** 正本はサーバーにあり、ローカルのダウンロード済みデータ（添付ファイル・全文本文）は**いつでも再取得できるキャッシュ**として扱う。以下のティア／期間／容量上限はすべて「キャッシュの保持・追い出しポリシー」であり、消えても機能は失われない（開けば再取得）。

「最新＝フル → 過去に向かって段階的に軽く」というグラデーションで、**新しいメールほど手厚く・古いメールほど軽く**ローカル保存する。段階は3ティア:

| ティア | 保存内容 |
|---|---|
| **フル** | 本文 + 添付ファイル（＋メタ） |
| **軽量** | 本文 + メタ（添付ファイルは削除。必要時に再DL） |
| **要約** | `clean_body`（引用除去済みの新規部分）+ メタ。重い `body_html_z`/`body_plain` を破棄。引用履歴はスレッドから再構成、全文はサーバー再取得（[THREADING.md](THREADING.md)） |

- **スター付きは全ティアで保護**（年齢に関係なくフル保持）。正本はサーバーなので破棄分はいつでも復元できる。
- **添付ファイルは全ティア共通でオンデマンド**（同期はメタ情報だけ保存し、本体は「開く／ダウンロード」時に初めてローカル保存）。したがって `full_window` は実質「**ダウンロード済みの添付を手元に何日残すか**」を決める。範囲外でも**最終アクセスから約3日は残す猶予**を設け、古い添付を一時的に開いた直後に消えないようにする（放置すれば次回適用で削除。§2 の遅延取得と整合）。

### 区切り方（2案。ブレないのは「時系列で区切る」から）

境界は**必ず時系列（新しい→古い）に沿って**引く。したがって新着が増えても境界は**なだらかに過去へ後退するだけ**で、既存メールがティア間を行き来するような**ブレは起きない**。

- **(A) 日数ベース（実装済み・既定）** — わかりやすく予測しやすい。
  - `full_window`: この期間内はフル保持。超えたら添付を削除（フル→軽量）。既定 `all`（非破壊）。
  - `body_window`: この期間を超えたら本文を要約に落とす（軽量→要約）。既定 `off`（非破壊）。
  - 容量上限（GB）は**保険**として維持し、期間内でも超過すれば古い添付から LRU で追い出す。
- **(B) 容量%グラデーション（設計案）** — 容量を上限いっぱいまで使い切りたい人向け。
  - 上限の **80%** を最新の**フルデータ**、続く **15%** で**添付を削除**、残り **5%** で**要約保存**に落とす。
  - **時系列で新しい順にバイトを充填**し、各しきい値に達した時点の日付が自動的に境界になる。データが増えると境界が過去へずれるだけで、しきい値（80/15/5）自体は固定なのでブレない。
  - (A) と併用可能（(B) で大枠を決め、(A) の期間で下限を保証する等）。将来オプションとして提供検討。

> ねらい: 「最新は快適に全部・古いものは容量に応じて賢く軽量化」を、ユーザーが**期間**でも**容量%**でも指定できるようにする。どちらも時系列区切りなので挙動は安定。

## 4. ウィンドウ外の検索

- ローカルは FTS5 で高速検索（ウィンドウ内 or メタデータ保持分）。
- **ウィンドウ外を探す時はサーバー検索（IMAP SEARCH）をオンラインで実行**するオプションを用意。ヒットしたものだけ取り込む。

---

## 5. 同期の流れ

```
初回: 選択ウィンドウ（例 6ヶ月）の範囲だけ取得 → 軽い初回同期
以降: 差分同期（UIDVALIDITY / UIDNEXT 管理）で新着を取得
随時: 本文/添付はオンデマンド、古いものは保持ポリシーに従い整理
```

## 6. データモデル（accounts に設定列を追加）

詳細は [DATABASE_SCHEMA.md](DATABASE_SCHEMA.md)。

- `accounts.sync_window`           — サーバー取得範囲 `'n50'|'3d'|'30d'|'3m'|'6m'|'all'` 等（既定 `'6m'`）
- `accounts.full_window`           — フル保持（本文＋添付）の期間 `'7d'|'30d'|'3m'|'6m'|'1y'|'all'`（既定 `'all'`＝非破壊）【実装済み・§3.5(A)】
- `accounts.body_window`           — 本文の全文保持の期間 `'off'|'3m'|'6m'|'1y'|'2y'`（既定 `'off'`＝非破壊）【実装済み・§3.5(A)】
- `accounts.storage_limit`         — ローカル容量上限（バイト。既定 2GB。保険＆%グラデーションの基準）【実装済み】
- `emails.body_compacted`          — 本文を要約保存に落とした印（`clean_body` のみ残存）【実装済み】
- `accounts.body_fetch`            — `'window'|'on_demand'`（既定 `'window'`）
- `accounts.attachment_fetch`      — `'on_demand'|'auto'`（既定 `'on_demand'`）
- `accounts.retention`             — `'window'|'metadata_only'|'keep_all'`（既定 `'window'`。§3.5 のティア導入で役割は縮小）

## 7. Tauri コマンド（抜粋）

| コマンド | 用途 |
|---|---|
| `account_set_sync_window` | 同期ウィンドウ変更（不足分の追加取得をトリガ） |
| `account_set_full_window` | フル保持期間の変更＋即時に保持ポリシー適用（§3.5(A)）【実装済み】 |
| `account_set_body_window` | 本文の全文保持期間の変更＋即時に要約適用（§3.5(A)）【実装済み】 |
| `storage_optimize` | 保持ポリシーを手動適用（古い添付削除＋本文要約＋容量保険）【実装済み】 |
| `account_set_retention` | 保持ポリシー変更（旧モデル） |
| `message_fetch_body` / `attachment_download` | 本文・添付のオンデマンド取得 |
| `server_search` | ウィンドウ外をサーバー側（IMAP SEARCH）で検索 |
