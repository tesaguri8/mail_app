# 同期範囲・保持期間（取得する期間の選択）

**ステータス:** 計画（実装未着手）
**結論:** **すべてを無条件に取り込むのではなく、取得する期間をユーザーが選べる**（アカウント単位）。本文・添付は段階的に取得して軽快さと容量を両立する。

関連: [DEVELOPMENT_PLAN.md](DEVELOPMENT_PLAN.md)（Phase 4）/ [CROSS_PLATFORM.md](CROSS_PLATFORM.md)（独立 IMAP 同期）/ [DATABASE_SCHEMA.md](DATABASE_SCHEMA.md) / [DATA_STORAGE.md](DATA_STORAGE.md)

---

## 1. 同期ウィンドウ（ユーザー選択）

アカウントごとに「どこまで遡って取り込むか」を選べる。

- 選択肢（例）: **過去 1ヶ月 / 3ヶ月 / 6ヶ月 / 1年 / 2年 / 全期間**
- **既定: 6ヶ月**（軽快さと実用性のバランス。`全期間` はオプトイン）
- 端末ごとに異なる設定でよい（[CROSS_PLATFORM.md](CROSS_PLATFORM.md) の独立 IMAP 同期と整合。各端末が自分のウィンドウで同期）。
- 変更時: ウィンドウを広げたら不足分を追加取得、狭めたら範囲外をローカルから整理（下記 保持ポリシー）。

---

## 2. 段階的取得（メタデータ → 本文 → 添付）

全文を一度に落とさず、必要なものから取得して軽量・高速に保つ。

| レイヤ | 取得方針 |
|---|---|
| **ヘッダ/メタデータ** | 同期ウィンドウ内は取得。スレッド再構築・フィルタ・一覧の基盤（軽い） |
| **本文（clean_body）** | ウィンドウ内は取得（既定）。または開いた時にオンデマンド取得も選択可 |
| **添付ファイル** | **遅延取得**（開いた時／明示ダウンロード時）。サイズ上限・自動DL条件は設定可 |

> これにより「メタデータは広く・本文/添付は必要分だけ」を実現し、初回同期と容量を抑える。

---

## 3. 保持ポリシー（retention）

ローカルにどこまで残すかを選べる。

- **`window`（既定）**: 同期ウィンドウ内のみ保持。範囲外はローカルから整理。
- **`metadata_only`**: 範囲外も**メタデータ（ヘッダ・索引）は保持**し、本文/添付だけ削除して容量節約（検索ヒットは残る）。
- **`keep_all`**: 取得したものは消さない（容量優先でないユーザー向け）。
- 整理した本文/添付は**必要時に IMAP サーバーから再取得**（サーバーが源 → ローカルは可変キャッシュ）。

---

## 3.5 ローカル保存（取り込みは全部・保持は2軸）

**ステータス:** 実装済み（`feature/storage-tiers`）。

**取り込みは常に全部**（`sync_window` の UI 選択は廃止）。メールクライアントとして最新が欠けるのは致命的なので、範囲は絞らない（初回のフル取得は安全上限あり）。

**基本の考え方＝ローカルはキャッシュ。** 正本はサーバーにあり、ローカルの重いデータ（添付ファイル・全文本文）は**いつでも再取得できるキャッシュ**。**メタ情報と要約（`clean_body`）は恒久保存**（軽く、プロダクトの「記録性」に整合）。重いデータだけを2軸で出し入れする:

| 軸 | 設定 | 挙動 |
|---|---|---|
| **添付ファイル** | `full_window`「添付を手元に残す期間」 | ダウンロード済み添付を期間内は保持。**古いものは実ファイル削除**（メタは残り再DL可）。最終アクセスから**約3日は猶予**。既定 `all`（非破壊）。 |
| **本文（全文）** | `body_window`「テキスト全文を確実に残す期間」 | 期間内は全文を**保証保持**。より古い本文は**容量上限を超えたときだけ**古い順で**要約**（`clean_body` だけ残し `body_html_z`/`body_plain` を破棄。全文はサーバー再取得＝メール本文の 🔄）。既定 `all`（要約しない・非破壊）。 |

- **スター付きは常に保護**（添付・本文とも）。
- **添付はオンデマンド**（同期はメタだけ保存。本体は「開く／ダウンロード」時に初めてローカル保存）。
- **容量オーバー時の段階**: 上限まで全文 → 超えたら古い本文を要約（要約は極小なので 5% の余白＝1GB なら約50MB で数万通分を吸収）。**それでも保証データ自体が上限+5% を超える**ときは、要約では吸収できないので設定 UI で**上限値を赤字警告**（上限を上げるか保持期間を短くする）。
- **古いメールのローカル削除（キャッシュ追い出し）は採用しない**: 要約＋メタは軽く恒久保存する方が「手元に恒久的に残る証跡（記録性）」に合う。重いデータだけをキャッシュとして管理する。

> 境界は**時系列（新しい→古い）**に沿うので、新着が増えても境界が過去へずれるだけでブレない。

---

## 4. ウィンドウ外の検索

- ローカルは FTS5 で高速検索（ウィンドウ内 or メタデータ保持分）。
- **ウィンドウ外を探す時はサーバー検索（IMAP SEARCH）をオンラインで実行**するオプションを用意。ヒットしたものだけ取り込む。

---

## 5. 同期の流れ

```
初回: 選択ウィンドウ（例 6ヶ月）の範囲だけ取得 → 軽い初回同期
以降: 差分同期（UIDVALIDITY / UIDNEXT 管理）で新着を取得
随時: 本文/添付はオンデマンド、古いものは保持ポリシーに従い整理
```

## 6. データモデル（accounts に設定列を追加）

詳細は [DATABASE_SCHEMA.md](DATABASE_SCHEMA.md)。

- `accounts.sync_window`           — 常に `'all'`（取り込み範囲の UI 選択は廃止。§3.5）【実装済み】
- `accounts.full_window`           — 添付を手元に残す期間 `'7d'|'30d'|'3m'|'6m'|'1y'|'all'`（既定 `'all'`＝非破壊）【実装済み】
- `accounts.body_window`           — テキスト全文を確実に残す期間（保証）`'3m'|'6m'|'1y'|'2y'|'all'`（既定 `'all'`＝要約しない）。これより古い本文は容量超過時に要約【実装済み】
- `accounts.storage_limit`         — ローカル容量上限（バイト。既定 2GB。超過で古い本文を要約。+5% 超で赤字警告）【実装済み】
- `emails.body_compacted`          — 本文を要約保存に落とした印（`clean_body` のみ残存）【実装済み】
- `accounts.body_fetch` / `attachment_fetch` / `retention` — 旧モデルの列（現在は未使用。将来整理）

## 7. Tauri コマンド（抜粋）

| コマンド | 用途 |
|---|---|
| `account_set_full_window` | 「添付を手元に残す期間」の変更＋即時に保持ポリシー適用【実装済み】 |
| `account_set_body_window` | 「テキスト全文を確実に残す期間」の変更＋即時に適用【実装済み】 |
| `storage_optimize` | 保持ポリシーを手動適用（古い添付削除＋容量超過時に本文要約）【実装済み】 |
| `mail_refetch` | 1通の全文をサーバーから再取得（要約の解除・本文キャッシュ復元）【実装済み】 |
| `attachment_download` | 添付のオンデマンド取得【実装済み】 |
| `server_search` | ウィンドウ外をサーバー側（IMAP SEARCH）で検索（将来） |
