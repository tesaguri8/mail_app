# スレッド再構築エンジン（引用解析・論理スレッド）

**ステータス:** 計画（実装未着手）／**コア機能**
**位置づけ:** 「チャット形式メール表示」を単なる見た目でなく、**独自のスレッド再構築**まで踏み込む目玉機能。
**解決したい課題:**
1. 「**返信**」で過去メールを再利用し、**件名は同じ（`Re: …`）なのに中身が全く別**のメールが増えている → 標準のスレッド表示が誤って同じ束にまとめる。
2. **返信の引用が長くてイライラ** → 読みたいのは「新しく書かれた部分」だけ。

関連: [FEATURE_SPEC.md](FEATURE_SPEC.md) / [DATABASE_SCHEMA.md](DATABASE_SCHEMA.md) / [UI_UX_DESIGN.md](UI_UX_DESIGN.md)

---

## 1. なぜ標準のスレッド分けが破綻するか

メールの返信には親メッセージを指すヘッダが付く:

- `Message-ID`: 各メール固有のID
- `In-Reply-To`: 返信元の Message-ID
- `References`: スレッドの Message-ID 連鎖

一般的なメールソフトは `In-Reply-To` / `References` ＋件名でスレッドを組む。
ところが「**内容は別件なのに古いメールに返信**」されると、ヘッダは古いスレッドを指したまま・件名も `Re: 同じ件名` のまま → **誤って同じスレッドに合流**してしまう。

→ ヘッダ依存が破綻するこの場面こそ、本エンジンが狙う領域。**中身（引用）を見て分ける／繋ぐ**。

---

## 2. 多層シグナルによる連結判定

単一の手がかりに依存せず、複数シグナルをスコアで統合する（上から優先）。

| 優先 | シグナル | 役割 |
|---|---|---|
| 1 | `Message-ID` / `In-Reply-To` / `References` | 各メールの確定ID。正常時の連結 |
| 1' | `Thread-Index` / `Thread-Topic`（Outlook/Exchange） | 会話ツリーをエンコードした独自ヘッダ。`References` 欠落時の補完 |
| 2 | **引用属性行の (from + 送信時刻)** | ヘッダが当てにならない時の連結／**分割の決め手** |
| 3 | **引用本文のフィンガープリント**（正規化ハッシュ） | 同じ履歴を引いているかを照合 |
| 4 | 件名の正規化（`Re:` `Fwd:` `RE：` `転送` 除去） | **弱い補助のみ**（過信しない） |

> 注: `Thread-Index` / `References` などヘッダ系シグナルは「返信で別件を送る」場合は**古い会話を指したまま**になる（同じ破綻をする）。よって**分割の最終判断は content 由来（優先2・3）が上書きできる**ように扱う。

### 引用属性行に (from + 時刻) が埋まっている

返信メールの引用部分の冒頭には、引用元の差出人・時刻が含まれることが多い:

```
On Mon, 30 Jun 2026 10:00:00 +0900, 山田太郎 <yamada@example.com> wrote:
2026年6月30日(月) 10:00 佐藤 <sato@example.com>:
-----Original Message-----
差出人: 山田太郎 <yamada@example.com>
送信日時: 2026年6月30日 10:00
```

この属性行から **(quoted_from, quoted_at)** を抽出し、手元の実メッセージの `From` ＋ `Date` と
**許容幅つきで突合**すれば、引用ブロック ↔ 元メッセージを結べる。
→ 「アドレス＋時間でIDを振る」発想は、この**引用突合の照合キー**として効く。

## 3. ID とグラフモデル（「スレッド＝IDの集合」）

```
各メール        = ノード（Message-ID ＋ from/to/sent/subject）
引用ブロック     = 「どの過去メッセージを引いているか」のエッジ（content由来）
ヘッダ参照       = In-Reply-To / References のエッジ（header由来）
   ↓
エッジで繋がる連結成分／分岐 ＝ アプリ独自の「論理スレッド」
```

- **論理スレッドはヘッダのスレッドと独立**に構築する。だから:
  - 件名が同じでも、**引用チェーンが切れ・新本文が無関係** → **別スレッドに分割**
  - 件名がバラバラでも、**引用が同じ履歴を指す** → **同じスレッドに結合**
- 各メールは 1 つの論理スレッドに所属（`emails.logical_thread_id`）。割り当ては `auto` / `manual` を区別。

---

## 4. 分割・結合ポリシー（自動分割＋手動上書き）

- **既定は自動**: ヒューリスティックのスコアで論理スレッドを自動構築。
  - 「件名同じ・参加者同じ」でも、**新本文の類似度が低く** かつ **引用が既存履歴を指さない** → 新スレッドへ**自動分割**。
- **ユーザーが上書き可能**:
  - 「このメールを別スレッドに分ける」「このスレッドと結合する」を手動操作。
  - `thread_assignment = 'manual'` のメールは再計算で動かさない（ユーザー意思を尊重）。
  - 手動操作は将来の判定改善のための材料にもする。
- **確信度の提示**: 自動分割は確信度を持たせ、UI で「別件かも」を控えめに示す。

---

## 4.5 アプリ内での再件名・整理

受信したメールの**元の件名はそのまま保持**しつつ、論理スレッドに**アプリ独自のタイトル**を付け直せる。

- **再件名（リネーム）**: 「`Re: 例の件` を `A社／6月見積もり` に」のように、スレッドへ分かりやすい題名を付与（`logical_threads.title`）。元の `Subject` は `auto_title` として残す。
- **同件名・別内容の整理**: 自動分割した各論理スレッドに別々のタイトルを付ければ、「件名は同じでも別案件」を**一覧上で明確に区別**できる。
- **ラベル/タグ**: 既存のタグ機能（[FEATURE_SPEC.md](FEATURE_SPEC.md) §2.3）と連携し、論理スレッド単位で分類。
- **手動の分割/結合**（§4）と合わせ、受信トレイを「自分が分かる単位」に組み替えられる。
- リネーム・分割・結合は `manual` 扱いとして保持し、再解析で勝手に戻さない。

> ねらい: メール提供元の件名付けに振り回されず、**アプリ内で自分の整理体系に作り替える**。これが「整理しやすさ」の核。

## 5. 表示（引用イライラの解消・SNS 風）

- 取り込み時に**引用・署名を解析して剥がし**、各メールの「**新しく書かれた部分（clean_body）**」を生成。
- チャットのバブルは clean_body のみを表示（自分＝右／相手＝左、時刻順）。
- 引用は**折りたたみ**、「引用を表示」で必要時のみ展開。

```
┌ 見積もりの件（論理スレ A）
│  相手 ▏ 来週の打ち合わせ可能ですか？
│  自分 ▕ 火曜の午後はいかがでしょう
└
┌ 見積もりの件（論理スレ B＝同じ件名・別内容）   ← 引用が別履歴なので自動分割
│  相手 ▏ 別件ですが請求書の宛名を…
└
```

- 横断検索（FTS5）は **clean_body** に張ると、引用の重複ヒットが消えて精度が上がる。

## 6. 解析パイプライン

```
受信(IMAP) → MIME解析(mail-parser)
   → 本文正規化（HTML→テキスト, 改行・空白正規化）
   → 引用/署名の分離（clean_body と quotes[] を抽出）
   → 各 quote から (quoted_from, quoted_at, fingerprint) を抽出
   → 連結判定（ヘッダ → 引用属性突合 → 引用fingerprint → 件名正規化）
   → 論理スレッド割当（auto。manual は固定）
   → DB 保存 ＋ FTS5(clean_body) インデックス
```

すべて**ローカルの Rust 側**（`src-tauri/src/services/parser/` ＋ `services/threading/`）で完結。SNS 中継のようなクラウド依存は不要。

## 7. 引用・署名の分離（実装の本体・難所）

メールの引用形式は多様。ヒューリスティックを作り込む必要がある。

- **形式の多様性**: plain の `>` 引用、HTML の `<blockquote>`、インライン返信（行間差し込み）、トップ/ボトムポスト。
- **属性行の言語差**: 日本語（`〜さんは（書）きました`, `差出人:` `送信日時:`, `-----元のメッセージ-----`）／英語（`On … wrote:`, `From: … Sent: …`, `-----Original Message-----`）の**パターン辞書**を用意。
- **署名・定型句**: `-- ` 区切り、`Sent from my iPhone`, 会社署名ブロック等の除去。
- **参考にする考え方**: Mailgun **talon**、GitHub **email_reply_parser** の手法。Rust で移植・自作。
- **時刻のゆらぎ**: タイムゾーン差・秒丸め → 突合は許容幅（例: ±数分）で行う。

> **100% にはならない前提**。スコア＋確信度で扱い、誤りは手動上書きで救済する（§4）。

## 7.5 活用できるヘッダ・メタデータ

メールヘッダには件名/本文以外にも、スレッド化・仕分け・信頼判定・解析に使える情報が多くある。
標準（RFC 5322 / 各拡張）と、送信側システムが付ける独自 `X-*` の両方が存在する。

### A. スレッド化に直接効く
| ヘッダ | 用途 |
|---|---|
| `Message-ID` / `In-Reply-To` / `References` | 返信連鎖の確定的な連結 |
| `Thread-Index` / `Thread-Topic`（Microsoft） | Outlook/Exchange が会話ツリーをエンコード。`References` 欠落時の補完 |

### B. 仕分け（自動分類）に効く
| ヘッダ | 用途 |
|---|---|
| `List-Id` / `List-Unsubscribe` / `List-Post` | **メルマガ・ML の判定**。個人メールと一括配信を分離（「仕分けが大変」の主因に効く） |
| `Precedence: bulk/list` / `Auto-Submitted: auto-*` | 自動送信・一括メールの検出（自動返信・通知の振り分け） |
| `X-Priority` / `Importance` | 重要度。ホーム/ウィジェットの強調に利用 |
| `Delivered-To` / `X-Original-To` | **自分のどのアドレス/エイリアス宛か** → マルチアカウント・用途別仕分け |
| `Return-Path` / `Sender` / `Reply-To` | 実際の送信元・返信先（なりすまし・転送の判別、返信先の最適化） |

### C. 信頼性・なりすまし検知
| ヘッダ | 用途 |
|---|---|
| `Authentication-Results` / `Received-SPF` / `DKIM-Signature` / `ARC-*` | SPF/DKIM/DMARC の認証結果。**差出人の正当性スコア**（「返信なりすまし」「別人化」の警告に有用） |
| `Received:`（経路チェーン） | 配送経路・原発時刻。発信元の推定 |

### D. 解析のヒント
| ヘッダ | 用途 |
|---|---|
| `X-Mailer` / `User-Agent` | 送信クライアント種別 → **引用/署名の形式推定**（例: iPhone Mail なら属性行パターンを切替）して §7 の解析精度を上げる |
| `Content-Type` / `Content-Language` | HTML/plain、本文言語 → 属性行パターン辞書（JP/EN）の選択 |

### E. 送信側システム独自の `X-*`（非標準・保証なし）
- CRM・問い合わせ管理・グループメール等が独自ヘッダを付けることがある（例: チケットID・キャンペーンID・配信元システム名など）。
- **存在すれば**案件IDとして強力な手がかりになるが、**送信側依存で保証されない**。あれば使う「ボーナス」として扱い、基盤は A〜D の標準ヘッダ＋引用解析に置く。

> 方針: 解析時にこれらを抽出して保存し、スレッド化（A・E）・自動仕分け（B）・信頼表示（C）・解析切替（D）に活用する。
> ただし**ヘッダ系は「返信で別件」に弱い**ため、最終的なスレッド分割は引用解析（§2 優先2・3）が上書きできる設計を保つ。

## 8. データモデル（要約）

詳細は [DATABASE_SCHEMA.md](DATABASE_SCHEMA.md)。

- `emails` に追加: `clean_body`（引用除去後の本文）, `body_fingerprint`, `logical_thread_id`, `thread_assignment`（auto/manual）, および活用ヘッダ（`thread_index` / `list_id` / `delivered_to` / `auth_result` / `precedence` / `x_mailer` 等）

- `message_quotes`: 1メール内の引用ブロック（`quoted_from` / `quoted_at` / `fingerprint` / `matched_email_id`）
- `email_fts`: 索引対象を `clean_body` に

## 10. フェーズ上の位置づけ

コア機能の一部として段階的に実装（[DEVELOPMENT_PLAN.md](DEVELOPMENT_PLAN.md)）:

- **Phase 4（メール同期・受信）**: 引用/署名分離・(from+時刻)抽出・fingerprint 生成までの解析基盤。
- **Phase 5（UI）**: clean_body のチャット表示・引用折りたたみ・論理スレッド表示。
- **Phase 7（検索・タグ）**: 自動分割の精緻化、手動分割/結合 UI、FTS5(clean_body) 連携。
